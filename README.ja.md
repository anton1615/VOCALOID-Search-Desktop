# VOCALOID Search Desktop

ニコニコ動画のVOCALOID動画を検索できる、モダンなSpotify風UIのローカルファースト・デスクトップアプリケーション。

> **[ニコニコ超検索](https://gokulin.info/search/) に触発されました** – 人気のニコニコ動画検索サービス。

これは [ウェブ版VOCALOID Search](https://github.com/anton1615/VOCALOID-Search) をTauriでローカルファーストのデスクトップアプリとして移植したものです。

> **注**: このプロジェクトは完全に **vibe coding** で構築されました – AIアシストによる反復開発で作成されています。

## 機能

- **モダンなインターフェース**: Spotify風のUI、プレイリスト形式のレイアウト
- **ライト/ダークモード**: テーマ切替対応
- **多言語対応**: 英語、日本語、中国語（繁体字）
- **PiPウィンドウ**: 常に最前面に表示されるピクチャー・イン・ピクチャープレイヤー
- **視聴履歴**: 視聴した動画の記録
- **視聴済み除外**: 検索結果から既に見た動画を除外
- **ウィンドウ状態の保存**: ウィンドウ位置、サイズ、最大化状態をセッション間で記憶
- **ローカルデータベース**: 動画データをローカルに保存し、オフラインで高速検索
- **カスタム数式での並べ替え・フィルタリング**: 再生数、マイリスト数、コメント数、いいね数を独自の重み付けでソート
- **オートスキップ**: 動画のエンディングを自動スキップ（エンドロールやスポンサー部分のスキップに便利）
- **埋め込みプレイヤー**: 公式ニコニコ埋め込みプレイヤーによる連続再生
- **キーワード+タグ検索**: タグフィルタ付き全文検索
- **無限スクロール**: 固定ページネーションではなく動的読み込み

## ウェブ版とデスクトップ版の比較

| 機能 | ウェブ版 | デスクトップ版 |
|------|---------|---------------|
| **デプロイ** | セルフホストサーバー (NixOS/Linux) | ローカルアプリ (Windows) |
| **稼働** | 24時間サーバー運用 | オンデマンド起動 |
| **スクレイパー** | systemdタイマーで自動実行 | 手動実行 |
| **マルチユーザー** | 対応（ユーザー登録可能） | 単一ユーザー、ローカルデータ |
| **モバイル対応** | PWA/TWAでAndroid対応 | 非対応 |
| **PiPモード** | コンパクトウィンドウにリサイズ | ネイティブ常時最前面ウィンドウ |
| **データ保存** | サーバー側データベース | ローカルファイルシステム |
| **オフライン検索** | サーバー接続が必要 | 完全ローカル |
| **プラットフォーム** | NixOS/Linux限定 | Windows (Linux対応予定) |

### どちらを選ぶべき？

**ウェブ版が適している場合:**
- 24時間自動スクレイピングを希望
- マルチユーザー対応が必要
- PWAでモバイルアクセスを希望
- NixOS/Linuxサーバーを運用している

**デスクトップ版が適している場合:**
- ネイティブデスクトップ体験を希望
- OSと密結合したPiPウィンドウを希望
- サーバー管理をしたくない
- 完全オフライン検索が必要

## 技術仕様

| レイヤー | 技術 |
|---------|------|
| **フロントエンド** | TypeScript, Vue 3, Vite, Tailwind CSS |
| **バックエンド** | Rust (Tauri 2.x) |
| **データベース** | SQLite (FTS5 全文検索) |
| **データソース** | [ニコニコ Snapshot API v2](https://site.nicovideo.jp/search-api-docs/snapshot.html), [GetThumbInfo API](https://site.nicovideo.jp/search-api-docs/thumb-info.html) |

### 実装アーキテクチャ

```
┌─────────────────────────────────────────┐
│           Vue 3 Frontend                │
│  (Search UI, Player, Settings, PiP)     │
└─────────────────┬───────────────────────┘
                  │ Tauri IPC
┌─────────────────▼───────────────────────┐
│           Rust Backend                   │
│  (SQLite, HTTP Client, File System)     │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│    Local SQLite Database (FTS5)         │
└─────────────────────────────────────────┘
```

### 主な技術的詳細

- **カスタムプロトコル**: HTTP localhostの代わりにTauriのカスタムプロトコル（`tauri://localhost`）を使用。ニコニコ埋め込みプレイヤーのドメイン制限を回避
- **FTS5全文検索**: SQLite FTS5でAND/OR/NOT演算子を使った高速なキーワード・タグ検索を実現
- **数式ベースのスコアリング**: 独自の重み付けで柔軟な並べ替えが可能

## 動作環境

- **OS**: Windows 10/11 (x64)
- **メモリ**: 最低4GB、推奨8GB
- **ストレージ**: 実行ファイル約10MB。データベースサイズは同期範囲による（例：VOCALOID キーワード、20日間、音楽カテゴリ ≈ 40MB）。WebView2 Runtime は通常 Windows 10/11 にプリインストール済み。
- **ネットワーク**: 動画再生とデータ同期にインターネット接続が必要

## データ保存先

データベースと設定は以下の場所に保存されます：
```
Windows: %APPDATA%\com.vocaloid-search.desktop
```

### ポータブルモード

ポータブルモード（データをアプリケーションフォルダに保存）を使用するには、実行ファイルと同じディレクトリに`data/`フォルダを作成してください：
```
<vocaloid-search-desktop.exeの場所>/data/
```

ポータブルモードが有効な場合、すべてのデータ（データベース、設定、サムネイル）がこのフォルダに保存されます。フォルダごと別のコンピュータにコピーすれば、同じデータを使用できます。

**注意**: ポータブルモードと標準モードの切り替えでは、データは自動的に移行されません。

## 使い方

### 利用の流れ

1. **起動時チェック**: アプリ起動時、データベースが空か古いかを自動チェックします。ニコニコSnapshot APIは毎日JST 5-6時頃に更新されます。データベースが古い場合、同期を促すメッセージが表示されます。

2. **データ同期**: 「データ同期」ページでスクレイパーの設定と実行：
   - **クエリ**: 検索キーワード（デフォルト: `VOCALOID`）
   - **最大日数**: 過去N日以内の動画のみ取得（デフォルト: 365、空欄で無制限）
   - **検索対象**: `tags`、`tagsExact`、`title`、`description`、または`tags,title`のような組み合わせ
   - **カテゴリ**: カテゴリフィルタ（デフォルト: `MUSIC`）
   
   「同期開始」をクリックして動画データをダウンロード。1日1回の同期をおすすめします。

3. **検索・閲覧**: 検索バーでキーワード検索。タグフィルタで結果を絞り込み。ソート数式で再生数やマイリスト数などの重み付けを調整。「視聴済み除外」を有効にすると、既に見た動画を非表示にできます。

4. **動画視聴**: 動画をクリックすると埋め込みプレイヤーで再生。リストから連続自動再生。再生設定でオートスキップを有効にすると、動画のエンディングを自動スキップできます。

5. **PiPモード**: ポップアウトボタンでPiPウィンドウを開く。ナビゲーションのない簡易プレイヤーで、常に最前面に表示。

### スクレイパー設定オプション

| オプション | デフォルト | 説明 |
|-----------|-----------|------|
| `query` | `VOCALOID` | ニコニコSnapshot APIの検索キーワード |
| `max_age_days` | `365` | 過去N日以内の動画のみ取得。空欄で無制限 |
| `targets` | `tags` | 検索対象: `tags`、`tagsExact`、`title`、`description`、または組み合わせ |
| `category_filter` | `MUSIC` | ニコニコカテゴリフィルタ（MUSIC, GAME, ANIME, ENTERTAINMENT, DANCE, OTHER） |

## ソースからのビルド

### 前提条件

- Node.js 18+
- Rust 1.70+ (`cargo`含む)
- Windows 10/11 SDK

### ビルド手順

```bash
# リポジトリをクローン
git clone https://github.com/anton1615/VOCALOID-Search-Desktop.git
cd VOCALOID-Search-Desktop/vocaloid-search-desktop

# 依存関係をインストール
npm install
```

#### 開発ビルド

```bash
npm run tauri dev
```

フロントエンド開発サーバーとTauriが開発モードで起動します。

#### 本番ビルド

```bash
# フロントエンドとバックエンドを一緒にビルド
npm run tauri build

# またはデバッグフラグ付きでビルド（ビルドが高速、バイナリは大きい）
npm run tauri build -- --debug
```

> **重要**: Tauriは埋め込みプレイヤー用にHTTP localhostの代わりにカスタムプロトコル（`tauri://localhost`）を使用します。これはニコニコの埋め込みプレイヤーが`localhost`ドメインからのリクエストを拒否するためです。
>
> **開発時の注意**: ニコニコがlocalhost接続を拒否するため、`tauri dev`で埋め込みプレイヤーをテストできません。フロントエンドの変更をテストするには、アプリケーション全体を再ビルドする必要があります。`--debug`を使用するとRustの最適化をスキップしてビルド時間を大幅に短縮でき、開発サイクルがスムーズになります。正式な配布には`--debug`なしのリリースビルドを使用してください。

ビルドされた実行ファイルの場所：
```
vocaloid-search-desktop/src-tauri/target/release/vocaloid-search-desktop.exe
```

## リリースノート

### v1.5.11 - 閲覧/再生境界の分離

**主な変更点:**
- Search / History / Watch Later を切り替えても、現在再生中の playback-bound list を上書きしないように修正
- 非再生ビューの復元は browsing state のみ更新し、現在の再生セッションを復活・クリアしないように統一
- メインウィンドウと PiP が同じ backend authoritative playback snapshot から player UI を更新するように統一

**バグ修正:**
- 非アクティブな list の refresh / restore が、現在再生中セッションに対して playback-cleared 副作用を発生させないように修正
- 再生イベント後にメインウィンドウと PiP が別々の window-local reset ルールへ依存しないように修正

**技術的な実装:**
- バックエンド：`set_browsing_list()` は `active_playback` を再バインドせず、browsing context のみ更新
- バックエンド：active playback clear は、active list が実際に invalidated された場合のみ `active-playback-cleared` を発行
- フロントエンド：メインウィンドウと PiP が共有 player orchestration を通じて authoritative playback refresh semantics を使用
- タブ切り替え、非アクティブ list の restore、二画面同期をカバーするテストを追加・更新
- delta specs を `playlist-context-management`、`unified-player-core`、新規 `playback-browsing-boundary` spec に同期

**メリット:**
- 別ページを閲覧しても現在の再生が中断されない
- 非アクティブ list の再読み込みで player が意図せずクリアされない
- 同じ backend playback event 後でもメインウィンドウと PiP の表示が一致する

### v1.5.10 - 検索再生スナップショット境界

**主な変更点:**
- 検索再生がページネーション全体で安定したメンバーシップを維持するため、視聴済み除外境界を凍結
- 履歴に不変の `first_watched_seq` を追加し、再生セッション中の一貫した「視聴済みを除外」動作を実現
- 手動スクロールとPiP連続再生が同じ凍結境界を使用するように統一
- 履歴とあとで見るの再生動作は変更なし - 検索のみスナップショット境界を使用

**バグ修正:**
- 「視聴済みを除外」が有効な場合、再生中に視聴済みマークされた動画がページ境界で突然検索結果から消える問題
- 検索ページネーションのメンバーシップが、各リクエストが最新の履歴状態を評価するため不安定だった

**技術的な実装:**
- データベース：履歴に `first_watched_seq` と `first_watched_at` カラムを追加（追加マイグレーション）
- データベース：`mark_watched()` が再視聴時に不変の初回視聴シーケンスを保持
- 状態：Search リストコンテキストバージョンに紐付いた `SearchPlaybackSnapshot` メタデータを追加
- コマンド：`set_playlist_index()` が検索からの再生時にスナップショットを作成/再利用
- コマンド：`execute_search()` がスナップショット有効時に凍結境界から視聴済み除外を解決
- 履歴境界、スナップショットライフサイクル、ページネーション動作をカバーする12のユニットテストを追加
- delta specs を `playlist-context-management`、`playlist-state-sync`、`video-search` specs に同期

**メリット:**
- 再生セッション中の安定した検索結果
- 動画を視聴済みマークしても予期しない動画の消失がない
- 手動スクロールとPiP再生の一貫した動作


### v1.5.9 - 検索ページネーション安定性

**主な変更点:**
- 同じ値が多いフィールド（例：いいね数）でのページネーションずれを修正
- SQL ORDER BY に決定論的 tie-breaker（`v.id`）を追加し、ページネーションを安定化
- `get_search_state()` が権威ある list context から同期された `results` を返すように修正

**バグ修正:**
- いいね数/再生回数で並び替えると、同じ値の動画がページ間でずれる可能性があった
- ページ境界（例：50番目 → 51番目の動画）で予期しないギャップや重複が発生
- `get_search_state()` が実際の list context と一致しない古い `results` を返す可能性

**技術的な実装:**
- バックエンド：すべての検索 ORDER BY 句に `, v.id <dir>` tie-breaker を追加
- バックエンド：`build_search_query()` と `execute_search()` が同じ安定した並び順を使用
- バックエンド：`get_search_state()` が `list_context.items` から `results` を同期
- tie-breaker 動作を検証する3つのユニットテストを追加
- delta specs を `video-search` spec に同期

**メリット:**
- ページ境界で一貫したプレイリストの並び順
- さらに読み込む際に予期しないギャップが発生しない
- PiP 再生がページ境界を越えても安定


### v1.5.8 - Load More 競合状態の修正

**主な変更点:**
- `loadMore()` と `search()` の間の競合状態を修正（51番目の動画でプレイリストにギャップが発生する問題）
- フロントエンドの `loadMore()` に `loading` 状態チェックを追加し、検索中の実行を防止
- `loadMore()` はローカルで追加せず、バックエンドから結果を同期するように変更
- バックエンドの `load_more` コマンドが `extend_list_context_items` の戻り値を検証

**バグ修正:**
- 並び替え/フィルタを素早く切り替えると、51番目以降の動画にギャップが発生
- `search()` が結果をクリアしている間に `loadMore()` が実行され、新旧の結果が混在

**技術的な実装:**
- フロントエンド：`SearchView.vue` の `loadMore()` の先頭に `loading.value` チェックを追加
- フロントエンド：`loadMore()` 成功後に `api.getSearchState()` を呼び出して完全な結果を同期
- フロントエンド：PiP の `playNext()` もメインウィンドウの loading 状態をチェック
- バックエンド：`extend_list_context_items` 失敗時（バージョン不一致）に `load_more` がエラーを返す
- delta specs を `playlist-context-management` spec に同期

**メリット:**
- 並び替えを素早く切り替えてもプレイリストの動作が一貫
- 50番目以降の動画でギャップが発生しない
- PiP再生はメインウィンドウの検索操作の影響を受けない


### v1.5.7 - レガシー状態フィールドの削除

**主な変更点:**
- Rustバックエンドからレガシー状態フィールド（`playlist_index`、`search_results`、`history_results`、`watch_later_results`、`playlist_type`）を削除
- 状態管理が`ListContext`モデルに統一されました
- ブラウジングコンテキスト追跡用の`set_browsing_list()`メソッドを追加
- すべてのclippy警告を修正し、コードをクリーンに

**技術的な実装:**
- `playlist_index` → `active_playback.current_index`
- `*_results`フィールド → `list_contexts[id].items`
